# 3단계: 로그인 / 보안 패치

> 난이도: 🟡 보통 | 소요: 1~2일 | **운영 전 필수**

## 현재 보안 취약점 (심각)

| 문제 | 위험도 | 설명 |
|------|--------|------|
| 비밀번호 평문 저장 | 🔴 | DB에 "1234" 그대로 저장됨 |
| URL 파라미터 조작 | 🔴 | `?role=operating` 붙이면 관리자 접근 가능 |
| localStorage 인증 | 🟡 | 탈취 시 세션 가로채기 가능 |
| RLS 미적용 | 🔴 | 아무 사용자가 모든 테이블 읽기/쓰기 가능 |

## 작업 순서

### 3-1. 비밀번호 해싱 (AI 작업)

```
현재 profiles 테이블에 비밀번호가 평문으로 저장되어 있어.
bcrypt 해싱으로 변경해줘.

변경:
1. 로그인 시 bcrypt.compare()로 비밀번호 검증
2. 회원가입/비밀번호 변경 시 bcrypt.hash()로 저장
3. 기존 평문 비밀번호를 일괄 해싱하는 마이그레이션 함수 작성
4. bcryptjs 패키지 설치 필요: npm install bcryptjs
```

### 3-2. Supabase Auth 전환 (AI 작업)

```
localStorage 기반 인증을 Supabase Auth로 변경해줘.

변경:
1. 로그인: supabase.auth.signInWithPassword()
2. 로그아웃: supabase.auth.signOut()
3. 세션 확인: supabase.auth.getSession()
4. App.jsx에서 AuthProvider 래핑
5. 각 대시보드에서 localStorage.getItem('user') 대신 useAuth() 훅 사용
6. 기존 profiles 테이블의 사용자를 Supabase Auth로 마이그레이션하는 스크립트 작성
```

### 3-3. RLS 정책 설정 (직접 + AI)

**직접 할 것:** Supabase Dashboard → Authentication → Policies

```
각 테이블에 RLS 정책을 설정하는 SQL을 작성해줘.

규칙:
- funeral_cases: 
  - admin → 모든 접수 읽기/쓰기
  - leader → 본인 배정 건만 읽기/수정
  - dealer → 본인이 접수한 건만 읽기
- settlements: admin만 쓰기, 본인 건만 읽기
- partners: admin만 승인/수정, 본인 프로필만 읽기
- profiles: 본인 프로필만 수정
- coupons: admin/dealer만 발행, 본인 발행 건만 읽기
- vendors: admin만 승인, 본인 업체만 수정
```

### 3-4. URL 접근 제한 (AI 작업)

```
현재 /admin 페이지에 ?role=operating 파라미터로 관리자 접근이 가능해.
이걸 서버 측에서 역할(role)을 검증하도록 변경해줘.

변경:
1. App.jsx의 라우팅에 ProtectedRoute 컴포넌트 추가
2. Supabase Auth 세션에서 role 확인
3. role이 'admin'이 아니면 /login으로 리다이렉트
4. URL 파라미터(role=operating)는 무시하고 DB에서 역할 조회
```

## 완료 확인

- [ ] 딜러가 /admin URL 직접 접속 시 차단되는가?
- [ ] RLS가 켜진 상태에서 각 역할별 데이터 접근이 정상인가?

## ⚠️ 주의
- 기존 사용자 비밀번호 마이그레이션 필요 (한 번만 실행)
- RLS 켜기 전에 충분히 테스트 (잘못 설정하면 모든 데이터 접근 차단됨)
- 테스트 계정으로 모든 역할(admin, leader, dealer, master) 로그인 확인
