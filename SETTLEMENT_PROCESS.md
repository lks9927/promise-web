# 💰 10년의 약속 - 정산 프로세스 명세서

> **이 문서는 장례 서비스 플랫폼 "10년의 약속"의 정산(Settlement) 비즈니스 로직을 정의합니다.**
> **모든 AI 및 개발자는 코드 수정 전 반드시 이 문서를 먼저 읽고 이해해야 합니다.**

---

## 1. 핵심 용어 정의

| 용어 | 방향 | 의미 |
|------|:----:|------|
| **수수료** | **+ (입금)** | 본사로 들어오는 돈. 팀장이 장례 진행 후 본사에 납부하는 사용료 |
| **커미션** | **- (출금)** | 본사에서 나가는 돈. 접수/관리에 대한 보상으로 파트너에게 지급 |

**본사 순이익 = 수수료(+) - 커미션(-) 합계**

---

## 2. 파트너 역할 구조

```
┌──────────────────────────────────────────────────────┐
│                    본사 (Admin)                       │
│                                                      │
│   수수료(+) 입금              커미션(-) 출금           │
│       ↑                          ↓                   │
├──────────────────────────────────────────────────────┤
│                                                      │
│  [장례 진행 쪽]              [장례 접수 쪽]            │
│                                                      │
│  ┌─────────────┐           ┌─────────────┐          │
│  │ 마스터 팀장  │           │ 마스터 딜러  │          │
│  │ (+) 수수료   │           │ (-) 커미션   │          │
│  │ (-) 관리커미션│           │ (-) 관리커미션│          │
│  └──────┬──────┘           └──────┬──────┘          │
│         │ 관리                    │ 관리              │
│  ┌──────┴──────┐           ┌──────┴──────┐          │
│  │ 일반 팀장   │           │ 일반 딜러   │          │
│  │ (+) 수수료   │           │ (-) 커미션   │          │
│  └─────────────┘           └─────────────┘          │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### 각 역할의 돈 방향 (본사 기준)

| 파트너 유형 | 입금 (+) | 출금 (-) | 설명 |
|------------|:--------:|:--------:|------|
| **딜러** (일반/마스터) | ❌ 없음 | ✅ 커미션 | 접수만 하므로, 본사가 대가를 지급 |
| **팀장** (일반) | ✅ 수수료 | ❌ 없음 | 장례 진행 후 본사에 사용료 납부 |
| **마스터 팀장** | ✅ 수수료 | ✅ 관리커미션 | 본인 진행 시 입금, 하위 팀장 관리 시 출금 |

## 3. 장례 한 건의 정산 흐름

### 3.1 접수 단계 (커미션 발생)

장례 건을 **누가 가져왔느냐**에 따라 4가지:

| # | 접수자 | 딜러 커미션 (-) | 마스터 관리커미션 (-) |
|---|--------|:--------------:|:-------------------:|
| A | 일반딜러 (마스터 있음) | ✅ 발생 | ✅ 발생 |
| B | 일반딜러 (마스터 없음) | ✅ 발생 | ❌ 없음 |
| C | 마스터 딜러 직접 접수 | ✅ 발생 (동일 금액) | ❌ 없음 (본인=마스터) |
| D | 본사 직접 접수 | ❌ 없음 | ❌ 없음 |

> **핵심 원칙**: 마스터든 일반이든 접수 커미션 금액은 **동일**하다.
> 관리 커미션은 "마스터가 일반 유저를 관리해주는 대가"이므로, 마스터≠본인일 때만 발생.

### 3.2 쿠폰 적용 여부 (×2)

위 4가지 각각에 대해:
- **쿠폰 미적용**: 위 정산만 발생
- **쿠폰 적용**: 위 정산 + **고객 캐시백** 정산 1건 추가 발생

→ **4 × 2 = 8가지 접수 시나리오**

### 3.3 진행 단계 (수수료 + 커미션 발생)

장례를 **누가 진행했느냐**에 따라 2가지:

| # | 진행자 | 팀장→본사 수수료 (+) | 마스터팀장 관리커미션 (-) |
|---|--------|:------------------:|:----------------------:|
| ① | 일반 팀장 (마스터 있음) | ✅ 발생 | ✅ 발생 |
| ② | 마스터 팀장 직접 진행 | ✅ 발생 | ❌ 없음 (본인=마스터) |

### 3.4 총 조합

**8가지 접수 × 2가지 진행 = 총 16가지 정산 시나리오**

## 4. 정산 항목 (Settlement Types)

하나의 장례 건에서 발생 가능한 정산 항목은 최대 5가지:

| 순서 | type (DB 값) | 방향 | 대상자 | 설명 |
|:----:|-------------|:----:|--------|------|
| 1 | `usage_fee_remittance` | **+** 입금 | 팀장 → 본사 | 장례 진행 사용료 |
| 2 | `dealer_commission` | **-** 출금 | 본사 → 딜러 | 접수 커미션 |
| 3 | `dealer_override` | **-** 출금 | 본사 → 마스터딜러 | 딜러 관리 커미션 |
| 4 | `leader_override` | **-** 출금 | 본사 → 마스터팀장 | 팀장 관리 커미션 |
| 5 | `customer_cashback` | **-** 출금 | 본사 → 고객 | 쿠폰 캐시백 |
| 6 | `additional_income` | **-** 출금 | 본사 → 팀장 | 부가수입 분배금 |

## 5. 자동 정산 트리거 (DB)

### 트리거 발동 조건
- `funeral_cases` 테이블의 `status`가 **`team_settling`**으로 변경될 때 자동 실행
- 함수명: `calculate_settlements()`
- 위치: DB Trigger (PostgreSQL / Supabase)

### 실행 로직 (순서대로)

```
1. funeral_cases.status → 'team_settling' 으로 변경됨

2. 역할 식별:
   - dealer_id → 접수 딜러
   - team_leader_id → 진행 팀장
   - partners 테이블에서 각각의 master_id 조회

3. 정산 생성 (모두 status='pending'):
   
   A. 팀장 수수료 (+)
      → usage_fee > 0 이면: usage_fee_remittance 생성
      → 세금: VAT 10% 추가
   
   B. 딜러 커미션 (-)
      → dealer_id가 있으면: dealer_commission 생성
      → 세금: 원천징수 3.3% 차감
      
      B-1. 딜러 쿠폰으로 고객 직접 접수한 경우
           → dealer_id가 NULL이어도 coupons.issued_by에서 딜러 자동 추적
   
   C. 마스터딜러 관리커미션 (-)
      → master_dealer_id ≠ dealer_id 이면: dealer_override 생성
      → 세금: 원천징수 3.3% 차감
   
   D. 마스터팀장 관리커미션 (-)
      → master_leader_id ≠ team_leader_id 이면: leader_override 생성
      → 세금: 없음 (사업자)
   
   E. 고객 캐시백 (-)
      → coupon_code가 있으면: customer_cashback 생성
      → 세금: 없음
```

## 6. 커미션 금액 정책 (commission_policies 테이블)

| 필드명 | 기본값 | 설명 |
|--------|:------:|------|
| `base_margin` | 1,000,000 | 기본 마진 |
| `customer_payback` | 100,000 | 고객 캐시백 금액 |
| `sales_dealer_regular` | 200,000 | 딜러 접수 커미션 (일반/마스터 동일) |
| `sales_dealer_master_override` | 50,000 | 마스터딜러 관리 커미션 |
| `sales_dealer_master_direct` | 200,000 | (예비) 마스터 직접 접수 시 커미션 |
| `sales_leader_regular` | 700,000 | (예비) 팀장 영업 커미션 |
| `sales_leader_master_direct` | 700,000 | (예비) 마스터팀장 직접 영업 커미션 |
| `exec_leader_master_override` | 50,000 | 마스터팀장 관리 커미션 |
| `exec_leader_master_direct` | 200,000 | (예비) 마스터팀장 직접 진행 시 |

> **"예비"** 표시 항목: 팀장이 직접 접수+진행하는 경우의 정산 방식이 아직 미결정 상태.
> 결정 후 트리거 함수에 반영 필요.

## 7. 세금 처리

| 대상 | tax_type | 계산 |
|------|----------|------|
| 팀장 수수료 입금 | `vat_10` | 금액 × 10% **추가** (본사가 받는 총액 증가) |
| 딜러 커미션 출금 | `withholding_33` | 금액 × 3.3% **차감** (실지급액 감소) |
| 마스터팀장 관리커미션 | `none` | 사업자이므로 원천징수 없음, 전액 지급 |
| 고객 캐시백 | `none` | 세금 없음 |

## 9. 정산 상태 (Settlement Status Flow)

```
pending (대기) → paid (지급완료) → completed (처리완료)
```

- **pending**: 정산 생성됨, 아직 송금/입금 전
- **paid**: 커미션 송금 완료 or 수수료 입금 완료
- **completed**: 관리자 최종 확인 완료
